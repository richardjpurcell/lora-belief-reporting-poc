# LoRa POC Run 003: CRC-Enabled Two-Transmitter Baseline

## Date

2026-06-22

## Purpose

This run tested the CRC-enabled version of the LilyGO LoRa proof-of-concept firmware using two transmitters and one receiver. The goal was to confirm that enabling LoRa CRC preserves clean packet delivery and eliminates the malformed payload observed in Run 001.

## Hardware

- RX: LilyGO LoRa32 T3 V1.6.1, running CRC-enabled receiver firmware
- TX-A: LilyGO LoRa32 T3 V1.6.1, sending logical node `N01`
- TX-B: LilyGO LoRa32 T3 V1.6.1, sending logical node `N16`
- Frequency: 915 MHz
- Antennas attached to all boards
- RX connected to laptop by USB
- TX boards powered separately or by USB

## Software

- Conda environment: `dcoss-lora-poc`
- Python: 3.11.15
- pyserial: 3.5
- pandas: 3.0.3
- Logger: `scripts/receiver_logger.py`
- Parser: `scripts/parse_receiver_log.py`

## Input and Output Files

- Raw receiver log: `logs/rx_run_003_crc_long.csv`
- Parsed valid packets: `logs/parsed_run_003_crc_long.csv`
- Malformed/rejected packets: `logs/parsed_run_003_crc_long_rejects.csv`

## Packet Format

Run 003 used the current packet format:

```text
RX,recv_ms,run_id,node_id,seq,tx_ms,region,event,priority,usefulness,stale_after,policy,rssi,snr
```

Example valid receiver row:

```text
RX,512009,R0,N01,87,443265,A,1,0.82,0.74,30,U,-47,9.50
```

The packet format does not yet include a separate physical transmitter identifier (`tx_id`). This should be added in the next firmware/schema revision.

## Interpretation

Run 003 is the first clean, meaningful proof-of-concept baseline. It shows that two LilyGO LoRa transmitters can send synthetic sensing packets with priority and usefulness metadata to one receiver, and that the receiver-side Python pipeline can produce a clean parsed dataset.

The result also confirms that CRC-enabled firmware produced no malformed packet rows in this longer run. This is an improvement over Run 001, where one malformed receiver row was observed and preserved in the reject file.

The two logical nodes delivered the same number of packets. Their delivered usefulness totals differ because the packet metadata assigns different synthetic usefulness values to the two streams. At this stage, that difference is hard-coded and should not be overinterpreted as a policy result. The main result is that the experimental logging and parsing pipeline now works.

## Known Limitations

- The transmitter and receiver `millis()` clocks are not synchronized. Receiver inter-arrival time is meaningful, but `recv_ms - tx_ms` should not yet be interpreted as true latency.
- Sequences continued from already-running transmitters rather than restarting at zero for the run.
- The packet format does not yet separate physical transmitter identity from logical sensing node identity.
- The run does not yet implement a real reporting-policy comparison.
- Usefulness and priority values are currently static payload fields rather than generated by a trace-selection process.

## Next Steps

1. Add a physical transmitter identifier (`tx_id`) to the packet format.
2. Update `scripts/parse_receiver_log.py` to parse the new 15-field schema.
3. Repeat a shorter clean run as `run_004_txid_crc`.
4. Begin the first minimal policy comparison, likely using two fixed streams with the same delivery opportunity but different synthetic usefulness values.
5. Later, replace static payload values with scheduled packets generated from a synthetic trace CSV.
